# Supabase migrations

> **Invariants**: every migration is forward-only by default, idempotent where possible, RLS-enabled in the same file as table creation, with indexes on every FK and policy column.

## The eight rules

### 1. Filename is `YYYYMMDDHHMMSS_<snake_case_description>.sql`

```
20260520143000_add_<resource>_table.sql      ✓
2026-05-20_add_<resource>.sql                 ✗  (wrong format)
add_<resource>.sql                            ✗  (no timestamp = no order)
20260520_add_<resource>.sql                   ✗  (date-only ambiguous; use full timestamp)
```

The timestamp is interpreted as TZ-naive — collisions handled by adjusting seconds.

### 2. Forward-only by convention

Don't write `down` migrations. If you need to revert, write a NEW migration that undoes the change. Reasons:

- Down migrations diverge from "what's actually in production" over time.
- They tempt mid-flight rollbacks that corrupt FK chains.
- You can always restore from PITR (Supabase point-in-time recovery) for genuine emergencies.

### 3. Idempotent where possible

Use `if not exists` / `or replace` / `drop ... if exists` so re-running the same migration on a partially-applied DB succeeds:

```sql
create table if not exists public.<resource>_tags (
  id bigint generated by default as identity primary key,
  <resource>_id bigint references public.<resource>(id) on delete cascade not null,
  tag_id bigint references public.tags(id) not null,
  unique (<resource>_id, tag_id)
);

create index if not exists idx_<resource>_tags_<resource>_id on public.<resource>_tags(<resource>_id);
create index if not exists idx_<resource>_tags_tag_id on public.<resource>_tags(tag_id);
```

Exceptions: data backfills are NOT idempotent without explicit guard. Use:

```sql
update public.<resource>
set <derived_field> = <computed>
where <derived_field> is null and <condition> = true;
```

The `where ... is null` makes the update a no-op on second run.

### 4. RLS in the same migration as the table

```sql
create table public.<resource>_blocks (
  id bigint generated by default as identity primary key,
  <resource>_id bigint references public.<resource>(id) on delete cascade not null,
  start_date date not null,
  end_date date not null,
  reason text,
  created_at timestamptz not null default now()
);

alter table public.<resource>_blocks enable row level security;

create policy "owner reads own blocks"
  on public.<resource>_blocks for select
  using (<resource>_id in (select id from public.<resource> where owner_id = (select auth.uid())));

create policy "owner inserts on own <resource>"
  on public.<resource>_blocks for insert
  with check (<resource>_id in (select id from public.<resource> where owner_id = (select auth.uid())));

create policy "owner deletes own blocks"
  on public.<resource>_blocks for delete
  using (<resource>_id in (select id from public.<resource> where owner_id = (select auth.uid())));
```

Never split table-creation and RLS into separate migrations. The gap is a leak window.

### 5. Index every FK + every RLS-policy column

```sql
-- FK indexes
create index idx_<resource>_blocks_<resource>_id on public.<resource>_blocks(<resource>_id);

-- The policy filters on parent.owner_id via subquery.
-- If parent.owner_id had no index yet, add it:
create index if not exists idx_<resource>_owner_id on public.<resource>(owner_id);
```

Without these, every read does a sequential scan. Dashboards die at 10k rows.

### 6. Use the right types

| Use case | Type |
|---|---|
| Surrogate ID | `bigint generated by default as identity` (preferred) or `uuid default gen_random_uuid()` (when user-facing) |
| FK to `auth.users` | `uuid` (matches Supabase) |
| Strings | `text` (avoid `varchar(N)` unless N is meaningful) |
| Money | `numeric(12, 2)` |
| Timestamps | `timestamptz` (ALWAYS — never `timestamp`) |
| Booleans | `boolean` |
| Documents / flexible data | `jsonb` (never `json`) |
| Geometry / coordinates | `point` or `geography(point, 4326)` (PostGIS) |
| Enums | `create type ... as enum (...)` (don't roll your own with check constraints) |

### 7. Soft delete by `deleted_at timestamptz`

For tables that need it:

```sql
alter table public.<resource> add column if not exists deleted_at timestamptz;
create index if not exists idx_<resource>_deleted_at_null on public.<resource>(id) where deleted_at is null;
```

Reads filter by `where deleted_at is null`. The partial index keeps queries fast.

### 8. Regenerate types after schema change

Once the migration is applied (locally or remote), regenerate the TypeScript types:

```bash
# Local (after supabase db reset / push)
npm run supabase:gen-types
# equivalent to:
# supabase gen types typescript --local > src/integrations/supabase/types.ts

# Or against remote project
supabase gen types typescript --project-id <ref> > src/integrations/supabase/types.ts
```

Commit the types diff alongside the migration. CI doesn't auto-regenerate — manual step.

## Audit checklist (use during review)

For every new migration:

- [ ] Filename `YYYYMMDDHHMMSS_<snake_case>.sql`, newer than all existing.
- [ ] No `down` block (forward-only).
- [ ] `if not exists` / `or replace` where possible.
- [ ] `enable row level security` on every new user-data table.
- [ ] At least one policy per `select / insert / update / delete` (or explicit "no public access" comment).
- [ ] Every policy uses `(select auth.uid())` — see `supabase-rls-pattern` skill.
- [ ] Indexes on every FK and every RLS-policy column.
- [ ] Right types: bigint/uuid for IDs, text, timestamptz, numeric, jsonb, deleted_at when applicable.
- [ ] Data backfill (if any) is guarded by `where ... is null` or equivalent.
- [ ] `src/integrations/supabase/types.ts` regenerated and committed.
- [ ] Migration runs locally without error (`supabase db reset`).

## Common mistakes

- Loose SQL files in `supabase/` root (`fix_*.sql`, `check_*.sql`). They are NOT migrations; they should be one-off ops scripts in `scripts/admin/` or a consolidated migration. Don't create this clutter.
- Renaming a column without updating `src/integrations/supabase/types.ts`. TypeScript fails at build time, but only if `strictNullChecks` is on — silent runtime errors otherwise.
- Skipping RLS "for the MVP". RLS is the security model; "for the MVP" means production launches without it.
- Migration that drops a column same release as code that still references it. Use a 2-step: (a) deprecate column code → release; (b) drop column → next release.
- Adding a `not null` column to a populated table without `default`. Will fail on rows that exist.

## Long-running operations (locking risk)

Some operations take a strong lock and block writes:

- `alter table ... add column not null` (lock + table scan)
- `alter table ... add foreign key` (lock + scan)
- `create index` (without `concurrently` blocks writes)

Pattern for safe schema change on a populated table:

```sql
-- Step 1 (this migration): add nullable, default
alter table public.<resource> add column if not exists <field> text;
update public.<resource> set <field> = <value> where <field> is null;

-- Step 2 (NEXT migration, after backfill verified): make non-null
-- alter table public.<resource> alter column <field> set not null;
```

For indexes on big tables, use `create index concurrently` — Supabase Pro plan supports it.

## Local workflow

```bash
# 1. Create migration file
supabase migration new add_<resource>

# 2. Edit the .sql file in supabase/migrations/

# 3. Apply locally + regen types
supabase db reset                  # nukes local + reapplies all migrations
npm run supabase:gen-types

# 4. Test in code
npm run dev

# 5. Commit
git add supabase/migrations/<file>.sql src/integrations/supabase/types.ts
git commit -m "feat(supabase): add <resource> table"

# 6. Push to remote when ready
supabase db push                   # CAREFUL: applies to remote project
```
