# i18n — source-of-truth pattern

> Imported by the `i18n` arthus-harness plugin.

## What this plugin gives you

| Artifact | Purpose |
|---|---|
| `scripts/check_i18n.mjs` | Parity validator. Run via `npm run i18n:check`. Skips silently until `src/locales/` exists. |
| `src/locales/pt-BR.json` | Source-of-truth locale (default — adjust via `harness.config.json`). |
| `src/locales/en.json` | Mirror locale, in strict parity with the source. |
| `.claude/skills/i18n-source-of-truth/SKILL.md` | Reviewer-loaded skill — Claude enforces the rules during code review. |

## Configuration

The source-of-truth locale defaults to `pt-BR.json`. Override:

**Option 1 — `harness.config.json`** (recommended for the project):

```json
{
  "i18n": {
    "source": "en.json"
  }
}
```

**Option 2 — env var** (one-off / CI override):

```bash
I18N_SOURCE=en.json npm run i18n:check
```

## Adding a new locale

1. Copy `src/locales/<source>.json` to `src/locales/<new-locale>.json`.
2. Translate values in place — keep keys identical.
3. Run `npm run i18n:check` — must pass with `✓ All locales in parity.`.
4. Wire the locale in your i18n library (e.g. `react-i18next` resources).

## Adding a new key

1. Add to the source-of-truth file first.
2. Add the same key to every other locale, translated.
3. Run `npm run i18n:check` to confirm parity.
4. Use the key in code: `t('your.new.key')`.

If you commit step 1 without step 2, `i18n:check` fails — that's the point.

## Why this pattern matters

Drift accumulates silently:

- Dev A adds a key in the source locale, forgets the others.
- Dev B sees the app fall back to the source string in another language.
- Bug reported weeks later, mid-release.

The pre-commit validator is the only countermeasure that scales with team size.

## Recommended library

This plugin doesn't pin a library. Common choices:

- **react-i18next** — full ICU, lazy-loading, namespace support. Most common for React apps.
- **next-intl** — Next.js-specific, server-component aware.
- **lingui.js** — compile-time message extraction; good for large catalogs.

Pick one, wire it to consume `src/locales/*.json` (or namespace files), then this validator becomes the gate that keeps locales in sync.
