# Supabase RLS pattern

> **Invariant**: RLS is enabled on every table that holds user data. Policies use `(select auth.uid())`. There is no exception in production.

## Why this skill exists

RLS bugs are silent: a missing policy or a wrong filter doesn't fail loud — it leaks data. In any multi-tenant app, one bad policy means tenant A reads tenant B's rows. The DB doesn't tell you; users discover it.

## The patterns we follow

### 1. RLS enabled on creation, in the same migration

Always:

```sql
create table public.<resource> (
  id bigint generated by default as identity primary key,
  owner_id uuid references auth.users(id) on delete cascade not null,
  -- ... other columns
);

alter table public.<resource> enable row level security;

create policy "owner reads own <resource>"
  on public.<resource>
  for select
  using (owner_id = (select auth.uid()));

create policy "owner inserts own <resource>"
  on public.<resource>
  for insert
  with check (owner_id = (select auth.uid()));
```

Never split table-creation and RLS into separate migrations — the gap window leaks.

### 2. The `(select auth.uid())` wrap

Always wrap `auth.uid()` in `(select ...)`. Postgres caches the SELECT once per query; bare `auth.uid()` re-evaluates per row.

```sql
-- BAD — re-runs auth.uid() per row
using (owner_id = auth.uid())

-- GOOD — postgres caches it
using (owner_id = (select auth.uid()))
```

This is a **performance** issue at scale, not a correctness one — but the Supabase advisor will flag it. Make it a habit.

### 3. Index every column that appears in a policy

A policy is a `WHERE` clause that runs on every query. Without an index, you do a sequential scan on every read. RLS without indexes is how dashboards die at 10k rows.

```sql
create index idx_<resource>_owner_id on public.<resource>(owner_id);
```

In the same migration as the policy. Never deferred.

### 4. Separate policies by operation

Don't combine select/insert/update/delete in `for all`:

```sql
-- BAD — too permissive, reasoning hard
create policy "owner all" on public.<resource> for all
  using (owner_id = (select auth.uid()))
  with check (owner_id = (select auth.uid()));

-- GOOD — explicit by op
create policy "owner reads own <resource>"   on public.<resource> for select using (owner_id = (select auth.uid()));
create policy "owner inserts own <resource>" on public.<resource> for insert with check (owner_id = (select auth.uid()));
create policy "owner updates own <resource>" on public.<resource> for update using (owner_id = (select auth.uid())) with check (owner_id = (select auth.uid()));
create policy "owner deletes own <resource>" on public.<resource> for delete using (owner_id = (select auth.uid()));
```

Reason: each op has different audit / compliance implications. A delete policy on a financial table needs different scrutiny than a select policy.

### 5. Service role bypasses; client never does

`service_role` JWT bypasses RLS by design. It belongs in **edge functions only** (`supabase/functions/*`). Never:

- In `src/integrations/supabase/client.ts` (always use anon key here)
- In a hook
- In a build step that runs in the browser

Edge functions read `SUPABASE_SERVICE_ROLE_KEY` from env. Client reads `VITE_SUPABASE_ANON_KEY`. They are **different keys** even though both are JWTs.

### 6. Multi-tenant: role-based access via subquery

When you have multiple personas (e.g., owner / collaborator / admin), do **not** route admin access through `service_role`. Check the role inside the policy via a subquery against your `profiles` (or equivalent) table:

```sql
create policy "admins read all <resource>" on public.<resource> for select
  using (
    exists (
      select 1 from public.profiles
      where user_id = (select auth.uid()) and role = 'admin'
    )
  );
```

The exists subquery is cached per query like `auth.uid()`. Index `profiles.user_id` and `profiles.role` if the table is large.

For ownership-via-parent (e.g., "user reads rows whose parent they own"):

```sql
create policy "user reads <child> on own <parent>"
  on public.<child> for select
  using (
    parent_id in (
      select id from public.<parent>
      where owner_id = (select auth.uid())
    )
  );
```

## Audit checklist (use during review)

When reviewing a migration / new table:

- [ ] `alter table ... enable row level security` is in the migration.
- [ ] Every multi-tenant column has a policy.
- [ ] Every policy uses `(select auth.uid())` — not bare `auth.uid()`.
- [ ] Every column in a policy has an index in the same migration.
- [ ] Policies are separate per `select / insert / update / delete`.
- [ ] No `using (true)` or `with check (true)` on user-data tables.
- [ ] Admin / role-based access is checked via subquery, not `service_role`.
- [ ] No client code (`src/`) imports `SUPABASE_SERVICE_ROLE_KEY`.

## Common mistakes

- Forgetting RLS on a join table (`<parent>_<child>`, `<resource>_tags`). Multi-tenant inheritance means every related table needs its own policy.
- Using `auth.role() = 'authenticated'` as the only filter. Authenticated ≠ authorised — every authenticated user sees everything.
- Policy that allows `select *` from `auth.users` (don't expose `encrypted_password`, `email`, etc. through join).
- Filter only on insert (`with check`) but not on select (`using`). Users can read what they couldn't write.
- Hard-deleting a row with active financial / audit trail. Use `deleted_at timestamptz` + partial index `where deleted_at is null`.

## Self-test

Before merging a migration, run:

```sql
-- as authenticated test user X
select * from <new_table>;          -- should return only X's rows
insert into <new_table> (...) values (...);  -- should succeed for X's data, fail otherwise
update <new_table> set ... where id = (other user's row);  -- should fail
delete from <new_table> where id = (other user's row);     -- should fail
```

Or use Supabase Dashboard → SQL Editor with role switcher.
