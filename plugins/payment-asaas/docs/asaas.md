# Asaas integration guide

> Imported by the `payment-asaas` arthus-harness plugin.

## What this plugin gives you

| Artifact | Purpose |
|---|---|
| `src/integrations/asaas/client.ts` | Env-driven Asaas client. 5s timeout, `AsaasError` class, error-code mapping. |
| `src/integrations/asaas/webhook-handler.ts` | Token validation + idempotency dispatch + log redaction. |
| `src/integrations/asaas/idempotency.ts` | Replay-safe processing. In-memory by default — swap for DB before prod. |
| `.claude/skills/asaas-integration/SKILL.md` | Reviewer-loaded skill — Claude enforces the 5 invariants on review. |

## Required environment variables

Set in `.env` (local) and your hosting provider's secret store (production):

| Variable | Purpose |
|---|---|
| `ASAAS_API_URL` | `https://sandbox.asaas.com/api/v3` for dev, `https://api.asaas.com/v3` for prod. |
| `ASAAS_API_KEY` | API key from Asaas dashboard. **Never** import in client code. |
| `ASAAS_MAIN_WALLET_ID` | Platform wallet for splits/fees. |
| `ASAAS_WEBHOOK_SECRET` | Token used to authenticate inbound webhooks. Set the same value in the Asaas webhook config. |

## Webhook setup

1. In the Asaas dashboard, register the webhook URL pointing at your handler (e.g. `https://api.example.com/webhooks/asaas`).
2. Set the **access token** field to the same string as `ASAAS_WEBHOOK_SECRET`. The handler rejects requests where `asaas-access-token` header doesn't match.
3. Subscribe to events you care about — typically:
   - `PAYMENT_CONFIRMED` — funds confirmed (cleared)
   - `PAYMENT_RECEIVED` — funds received in the wallet
   - `PAYMENT_REFUNDED` — refund completed
   - `PAYMENT_OVERDUE` — billing slip past due
4. Test the webhook in the dashboard — Asaas sends a synthetic event you can trace.

> **Common gotcha**: The dashboard says "webhook is active" once you click "save", but Asaas only delivers events to URLs registered under `/v3/webhooks` (the dedicated endpoint, not the older notification config). Verify by checking the request count in the dashboard after creating a test payment.

## Signature verification

Asaas currently uses a shared-secret token in the `asaas-access-token` header. The handler in `webhook-handler.ts` checks this on every request and returns 401 if it doesn't match.

When Asaas adopts HMAC signing, swap the token check for an `x-asaas-signature` HMAC-SHA256 verification using `ASAAS_WEBHOOK_SECRET`. The skill file has the upgrade snippet ready to paste.

## Idempotency rules

- `idempotency.ts` ships an in-memory `Set` — fine for local dev / unit tests.
- **Before going to production**, replace it with a Postgres-backed implementation:

```sql
create table public.asaas_webhook_events (
  id bigint generated by default as identity primary key,
  payment_id text not null,
  event text not null,
  processed_at timestamptz not null default now(),
  unique (payment_id, event)
);
```

Then use `insert ... on conflict (payment_id, event) do nothing` — first writer wins, replays are no-ops.

## Server-side amount

Never accept the amount from the client. Always read the price from your DB and compute the total server-side. See the skill file for the full pattern.

## Error mapping

`mapAsaasError(code)` in `client.ts` returns user-facing pt-BR copy. Add new codes there as you encounter them. Log the raw code server-side with a request ID; show the friendly message to the user.

## Log redaction

`redactForLog()` in `webhook-handler.ts` scrubs CPF, card number, and email before logging. If you wire Sentry / Datadog, configure their PII scrubbing on the same fields too — defense in depth.
